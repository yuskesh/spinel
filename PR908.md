# PR #908 Performance Investigation

## Summary

PR #908 (`234de48`, "fix(analyze,codegen): logical operators return operand values") fixed Ruby semantics for logical operators in value position. In Ruby, `&&` and `||` do not return booleans in general; they return one of their operands:

```ruby
1 && 2      # => 2
nil && 2    # => nil
false || 4  # => 4
true || nil # => true
```

The PR implemented that semantic change by teaching analysis and codegen to infer and emit value-preserving results for `AndNode` and `OrNode`. That is correct for expression/value contexts, but the same machinery is now used in condition contexts such as:

```ruby
if a && b
  ...
end

return false if a && b
```

In a condition context, the result value of `a && b` is not needed. Only its truthiness matters. PR #908 currently materializes the value anyway, often as `sp_RbVal`, then immediately calls `sp_poly_truthy` on it. This adds temporaries, branches, boxing, and dynamic truthiness calls on hot paths. The optcarrot FPS regression appears to come from this extra work.

## Small Reproduction

This small program isolates the relevant shape:

```ruby
class Probe
  def initialize
    @text = nil
    @n = 0
  end

  def attach
    @text = "hi"
  end

  def guard
    if @text && @text.length == 2
      @n += 1
    end
    @n
  end

  def value
    @text && @text.length
  end
end

p = Probe.new
puts p.guard
p.attach
puts p.guard
puts p.value
```

Generated C before PR #908 for `guard`:

```c

    SP_GC_SAVE();
    SP_GC_ROOT(self);
    if (((self->iv_text != NULL) && (sp_str_length(self->iv_text) == 2LL))) {
      self->iv_n += 1LL;
    }
    return self->iv_n;
  return 0;
}
```

Generated C after PR #908 for `guard`:

```c
static inline mrb_int sp_Probe_guard(sp_Probe *self) {
    SP_GC_SAVE();
    SP_GC_ROOT(self);
    const char * _t1 = self->iv_text;
    sp_RbVal _t2 = sp_box_nil();
    if ((_t1 != NULL)) {
      _t2 = sp_box_bool((sp_str_length(self->iv_text) == 2LL));
    } else {
      _t2 = ((void)(0), sp_box_nil());
    }
    if (sp_poly_truthy(_t2)) {
      self->iv_n += 1LL;
    }
    return self->iv_n;

```

This is the key issue. The condition `@text && @text.length == 2` does not need the Ruby value of the `&&` expression. It only needs a C truth value. Before PR #908, it compiled directly to a C `&&`. After PR #908, it constructs a Ruby-level value (`nil` or `bool`), boxes it, and converts it back to truthiness.

For the `value` method, PR #908's behavior is semantically useful:

Before PR #908:

```c

    SP_GC_SAVE();
    SP_GC_ROOT(self);
    return ((self->iv_text != NULL) && ((sp_str_length(self->iv_text)), TRUE));
  return FALSE;
}
```

After PR #908:

```c
static inline sp_RbVal sp_Probe_value(sp_Probe *self) {
    SP_GC_SAVE();
    SP_GC_ROOT(self);
    const char * _t3 = self->iv_text;
    sp_RbVal _t4 = sp_box_nil();
    if ((_t3 != NULL)) {
      _t4 = sp_box_int(sp_str_length(self->iv_text));
```

That is the behavior needed for value context. The problem is that condition context pays the same cost.

## Root Cause in the Compiler

The relevant code is in `spinel_codegen.rb`.

`compile_cond_expr` starts by compiling the predicate as a normal expression:

```ruby
def compile_cond_expr(nid)
  expr = compile_expr(nid)
  t = infer_type(nid)
  # ...
end
```

For `AndNode`, `compile_expr` now uses value-preserving emit:

```ruby
if t == "AndNode"
  left_nid = @nd_left[nid]
  right_nid = @nd_right[nid]
  lt = infer_type(left_nid)
  rt = infer_type(right_nid)
  at = and_result_type(nid)
  left_tmp = new_temp
  emit("  " + c_type(lt) + " " + left_tmp + " = " + compile_expr(left_nid) + ";")
  kind = logical_truthiness_kind(left_nid, lt)
  # ...
end
```

This is appropriate when the caller needs the value of the logical expression. It is too expensive when the caller only needs a condition.

The analyzer mirrors the semantic change by changing `AndNode` inference from `"bool"` to `and_result_type(nid)`:

```ruby
if t == "AndNode"
  return and_result_type(nid)
end
```

That is correct for expression types, but it means condition lowering sees some `&&` predicates as `poly` or nullable and wraps them dynamically.

## Why a Simple Revert Is Not Correct

Reverting PR #908 would restore performance, but it would also reintroduce incorrect Ruby semantics for logical expressions in value position.

The new test from PR #908 captures this:

```ruby
assert_equal(2, 1 && 2)
assert_equal(nil, nil && 2)
assert_equal(nil, false || nil)
assert_equal(false, false && 3)
assert_equal(true, true || nil)
assert_equal(nil, true && nil)
assert_equal(4, false || 4)
```

The compiler needs two different lowering modes:

1. Value mode: preserve Ruby operand values.
2. Condition mode: preserve only truthiness and short-circuiting.

PR #908 effectively uses value mode too often.

## Recommended Fix

### 1. Add condition-context logical lowering

Add a dedicated path for `AndNode` and `OrNode` inside `compile_cond_expr` before calling general `compile_expr`.

Conceptually:

```ruby
def compile_cond_expr(nid)
  if nid >= 0 && @nd_type[nid] == "AndNode"
    return compile_and_cond_expr(nid)
  end
  if nid >= 0 && @nd_type[nid] == "OrNode"
    return compile_or_cond_expr(nid)
  end

  expr = compile_expr(nid)
  # existing logic...
end
```

For simple cases, `compile_and_cond_expr` can return:

```c
(<left truthy expr> && <right truthy expr>)
```

For `OrNode`:

```c
(<left truthy expr> || <right truthy expr>)
```

This restores the pre-#908 shape for condition-only predicates.

### 2. Preserve short-circuiting when subexpressions emit statements

Some `compile_expr` calls emit setup statements before returning an expression. A naive:

```ruby
left = compile_cond_expr(left_nid)
right = compile_cond_expr(right_nid)
return "(" + left + " && " + right + ")"
```

can be unsafe if compiling the right side emits C statements before the returned expression. In that case, the right side's emitted statements would run before the C `&&` expression, breaking short-circuit semantics.

A robust implementation should either:

- only use inline `(<left> && <right>)` when both sides are known expression-only, or
- emit statement-level short-circuit code:

```c
mrb_bool _t = FALSE;
if (<left condition>) {
  _t = <right condition>;
}
```

For `||`:

```c
mrb_bool _t = TRUE;
if (!(<left condition>)) {
  _t = <right condition>;
}
```

This still avoids `sp_RbVal` boxing and `sp_poly_truthy` on a boxed result. It may be slightly larger than a raw C `&&`, but much cheaper than materializing Ruby operand values.

### 3. Add bool-only fast paths in value mode

Even in `compile_expr`, when both operands and result are `bool`, a value-preserving `&&` has the same value as a C boolean expression.

Safe fast path:

```ruby
if lt == "bool" && rt == "bool" && at == "bool"
  return "(" + compile_expr(left_nid) + " && " + compile_expr(right_nid) + ")"
end
```

Again, this should account for emitted statements. If either side can emit prologue statements, use the statement-level `mrb_bool` temp form instead.

This would help cases like:

```c
return (sp_Optcarrot_APU_Oscillator_active_p(...) && self->iv_valid_freq);
```

which currently becomes:

```c
mrb_bool _t215 = sp_Optcarrot_APU_Oscillator_active_p(...);
mrb_bool _t216 = FALSE;
if (_t215) {
  _t216 = self->iv_valid_freq;
} else {
  _t216 = _t215;
}
return _t216;
```

### 4. Keep value-preserving emit for real value contexts

Do not remove `and_result_type`, `or_result_type`, or the `sp_RbVal` materialization entirely. They are needed for cases like:

```ruby
x = maybe_string && maybe_string.length
return false || object
array << (a && b)
```

The optimization should be context-sensitive, not a semantic rollback.

## Risk Notes

The main risk is preserving short-circuiting when the right-hand side has side effects or when codegen emits setup statements while compiling it.

Unsafe transformation:

```ruby
left = compile_cond_expr(left_nid)
right = compile_cond_expr(right_nid)
return "(" + left + " && " + right + ")"
```

This can be wrong if compiling `right_nid` emits statements before returning the expression string. Those statements would be emitted before the final C `&&` and would execute even when the left side is false.

Safer default:

```c
mrb_bool tmp = FALSE;
if (left_condition) {
  tmp = right_condition;
}
```

This keeps Ruby short-circuit semantics and avoids Ruby-value materialization.

## Bottom Line

PR #908 fixed an important semantic bug, but it blurred value context and condition context. The FPS regression is caused by condition-only logical expressions being lowered as value-producing Ruby expressions.

The best fix is not to revert PR #908. Instead, add condition-context `AndNode` / `OrNode` lowering and bool-only fast paths so hot predicates compile back to cheap C truthiness checks while value contexts keep the correct Ruby operand-returning behavior.

## Update (2026-06-02): the recommended fix was implemented and it FAILED to improve performance

The recommendation above was implemented in full and **it does not recover the FPS — it makes optcarrot slower.** Recording this so the approach is not retried.

### What was implemented

Both recommended pieces, in `spinel_codegen.rb` (patch saved at `/tmp/p908/point1plus3.patch`, reverted from the tree):

1. **Condition-context lowering** — `compile_cond_expr` routes `AndNode`/`OrNode` through a new `compile_logical_cond` that emits a plain short-circuit C truth value (inline `(L && R)` when the right operand is expression-only; a guarded `mrb_bool` temp when it emits setup statements). No `sp_RbVal`, no `sp_poly_truthy`.
2. **Bool-only value fast path** — `compile_bool_logical_fast`: when result and both operands are `bool`, emit inline `(L && R)` instead of the branchy temp.

It is **correct**: self-host stage-2 clean, optcarrot checksum 59662, `test/logical_value_return.rb` passes, short-circuit preserved (verified with a side-effecting operand). It produces **exactly** the "before908" C shown above — verified line-for-line on `Oscillator#active?`, `Pulse#active?`, and `Holder#active?` (boxing gone, inline `&&` restored).

### The measurement (the part the original investigation skipped)

`perf stat` with ≤2 events is deterministic to ±0.03% on this workload; FPS is noisy (layout / frequency-cap / contention). Measuring **work done**, not just FPS:

| build | instructions | IPC | cycles | fps |
|-------|-------------:|----:|-------:|----:|
| point1+3 (the fix) | 4.611B | **2.15** | 2.15B | ~524 |
| clean HEAD | 4.585B | **2.25** | 2.04B | ~560 |

The fix **regresses ~6%**, rock-steady across 10 rounds. It is **not** "boxing is free": instructions barely move (+0.6%). It is a **−4% IPC drop** — a branch-prediction / i-cache effect.

### Why the premise does not hold at HEAD

`perf record` + generated-C diff put the regression in the **PPU hot loop** (`main_loop`, `render_pixel`), and the surprise is there:

- **`main_loop` has zero `sp_poly_truthy`; `render_pixel` has 5.** The hot path is barely boxed. The heavy boxing this document analysed lives in the **cold APU `active?` methods**.
- So PR #908's hot-path change was mostly **inline `&&` -> bool branchy-temp**, not boxing. Example, the `main_loop` skip-dot test `scanline == -1 && odd_frame`: clean HEAD emits an 8-line `mrb_bool` branchy-temp; the fix emits clean `if (((scanline==-1) && odd_frame))`.
- **At HEAD that inline form has worse IPC than the branchy-temp.** before908(inline)=608 > after908(branchy-temp)=564 — inline was faster *then*. But producing the inline shape at HEAD gives ~524 < clean HEAD's branchy-temp ~560. **The microarchitecturally-optimal form flipped** between before908's codebase and today's. The current (branchy-temp) code is already the faster form; restoring inline is pessimal.

Conclusion: PR #908's regression is real, but it is **not recoverable by a targeted logical-op codegen change** — the documented fix regresses it further. The boxing the document identified is in cold code, not the hot loop.

## Update (2026-06-02): PR #908 is not the only regression

Bisecting the whole optcarrot drop with **deterministic instruction count** (not just FPS) across buildable anchors (many commits in this window can't compile today's optcarrot pack — the malformed `sp_Tuple_int_int_int?`, fixed at `b481894` — so each was checksum-verified):

| commit | date | fps | instructions |
|--------|------|----:|-------------:|
| 5cf5c22 | 05-26 12:51 | 673 | 3.675B |
| b481894 | 05-26 22:16 | 651 | 3.680B |
| 79b303d | 05-27 end | 571 | 3.915B |
| HEAD | 06-02 | 562 | 4.585B |

- **The FPS damage is concentrated on 05-27** (651 -> 571, **-80 fps**) — the day of PR #908.
- It is **two mechanisms**: a modest real-work increase (+235M instructions on 05-27) and, dominantly, **layout/IPC** — `5cf5c22 -> b481894` lost **-22 fps with zero instruction change** (3.675B -> 3.680B). The 05-27 String / Regexp / pattern-match **feature wave** (e.g. `tr_s`, `partition`, regex backref, pattern-match `as`) bloats the binary and shifts the PPU hot loop's i-cache / branch layout without adding hot-path work.
- PR #908 (-38 fps) is the single largest cliff inside 05-27, but the surrounding feature commits account for the rest (-16 in `b481894..1c02519`, -20 in `1c02519..3afbdc4`).
- **05-28 -> HEAD added +670M instructions for only -9 fps** — that growth is cold new-feature code, not the hot loop.

So the regression is **layout-dominated and diffuse**: dozens of unrelated feature commits each nudging the hot loop's microarchitecture, plus PR #908's form change (also layout/IPC). There is no single recoverable culprit, and the targeted logical-op fix recommended above is counterproductive at HEAD. The real lever would be whole-binary layout control (function alignment/ordering / PGO — PGO previously showed no gain here), not the logical-op lowering.
